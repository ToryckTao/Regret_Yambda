# 0505 实验记录：session-run 切分、离线 SID 训练与后悔池评估

记录范围：从“重新设计数据集切分”开始，到 `replay_val/replay_test` 后悔池评估结束。

当前主线不再继续依赖 `user_response` simulator。原因是前面多轮实验显示 simulator 很难同时学会 listen/play/like/dislike/unlike 等多动作反馈，并且 MSE/MAE 与分类型指标都不能稳定打赢均值基线。因此本阶段转为：先把真实日志构造成更清楚的 MDP-like transition，再在 logged transitions 上训练离线 actor/critic，并用 replay 方式验证后悔池是否真的改变策略生成。

## 1. 后悔池到底对策略生成有什么影响

后悔池不是重新训练一个模型，也不是直接把 reward 加到最终推荐分数上。它是在策略生成 SID token 时，对 actor 的 logits 做 inference-time intervention。

当前 actor 生成一个 item 时，不是直接预测 dense item id，而是预测这个 item 的分层语义 ID，也就是 SID path，例如：

```text
item A -> SID = [12, 47, 5]
```

如果用户历史上对某些语义路径产生了负反馈，例如低播放、dislike、unlike，这些失败路径会进入该用户自己的 `RegretMemoryPool`：

```text
user u 的后悔池:
  ([12, 47, 5], phi=1.2, type=unlike)
  ([8, 31, 9], phi=0.8, type=low_play)
```

推理时，actor 原本在第 l 层会输出：

```text
logits_l = sid_head_l(context)
```

加入后悔池后，代码里的 `get_sara_logits()` 会计算候选 token 路径与后悔池路径的 prefix overlap。如果当前候选路径越像历史失败路径，就在对应 token 的 logit 上减去更大的 penalty：

```text
logits_l = logits_l - sara_eta * penalty_l
```

所以它对策略生成的实际影响是：

```text
相似历史失败路径的 SID token 概率下降
不相似路径基本不受影响
```

一个具体例子：

```text
后悔池里有失败路径 [12, 47, 5], phi=1.2

actor 当前生成候选：
  level 1 候选 token = 12
  level 2 候选 token = 47
  level 3 候选 token = 5

因为 prefix 高度重合，level 2/3 会得到更大的 penalty。
最后 softmax 后，这条路径的概率下降。
```

这就是图里 RAPI 的思想：把用户历史 regret semantic path 作为 soft mask 注入层级策略生成。当前代码里的符号对应关系是：

```text
B_regret        -> RegretMemoryPool
z^j_{1:L}       -> 后悔池里的 SID path
Phi_j           -> regret_strength/reward 计算出的 phi，并随 replay 更新按 gamma 衰减
Sim(prefix, z)  -> 代码里的 prefix one-hot overlap
eta             -> sara_eta，默认 0.5
D_{z_l=k}       -> penalty_l
```

最终 replay 评估结果证明这个机制确实在起作用：

| split | all_n | neg_n | neg_suppression | pos_suppression | ratio |
|---|---:|---:|---:|---:|---:|
| replay_test | 488,801 | 165,094 | 0.06896 | 0.02351 | 2.93x |
| replay_val | 466,762 | 158,424 | 0.06714 | 0.02201 | 3.05x |

这里的 suppression 定义是：

```text
suppression = base_logp(target_path) - pool_logp(target_path)
```

它越大，说明加入后悔池后，真实 target SID path 的概率被压得越低。负样本上的 suppression 约为正样本的 3 倍，说明后悔池不是无差别降权，而是更强地压制历史 regret 相关路径。

需要注意：这证明的是“干预机制符合设计预期”，还不能直接证明“最终推荐 reward 提升”。因为当前没有可信 user simulator，actor 选了一个新 item 后，真实用户会如何反馈是不可观测的。

## 2. 当前三个 loss 是否按图中 Joint Optimization 实现

结论：方向相似，但不是严格按照图中标准 Actor-Critic 公式实现。

图中公式的意思是：

```text
Critic:
L_v(phi) = E[(V_phi(s_t) - Q_hat(s_t, a_t))^2]

Actor:
L_pi(theta) = E[- A'_t * log pi_theta(action | state) - entropy]

Joint:
min L_v + L_pi
```

它依赖一个强化学习式的 advantage `A'_t`，并假设能得到策略动作对应的 value/advantage。

当前 `scripts/06_train_sid_offline.py` 的实际实现是离线 logged-transition surrogate，不是完整在线 Actor-Critic。代码里的 loss 是：

```text
critic_loss = MSE(q_pred, logged_reward)

pos_loss = 对 reward >= threshold 的 logged SID 做 NLL 行为克隆
neg_loss = 对 reward < threshold 的 logged SID 做 avoid loss，即 -log(1 - p(logged_token))
entropy = SID token 分布熵

actor_loss = bc_weight * pos_loss
           + avoid_weight * neg_loss
           - entropy_weight * entropy

total_loss = actor_loss + critic_weight * critic_loss
```

所以可以这样理解当前的三个核心 loss：

```text
1. 正样本模仿 loss：好反馈的 item，尽量学会生成它的 SID path。
2. 负样本规避 loss：坏反馈的 item，尽量降低它的 SID path 概率。
3. critic reward loss：用 actor 的 context 去拟合当前 logged reward。
```

它和图中公式的对应关系：

| 图中概念 | 当前实现 | 是否严格一致 |
|---|---|---|
| `L_v` critic value loss | `MSE(q_pred, reward)` | 部分一致 |
| `L_pi` actor policy loss | 正样本 BC + 负样本 avoid + entropy | 不严格一致 |
| `A'_t` regret 修正 advantage | 当前没有显式 advantage，只用 logged reward 的正负和强度做权重 | 不一致 |
| joint optimization | actor/critic 同时更新，总 loss 相加 | 部分一致 |

因此论文写法上需要谨慎：不能说当前代码已经实现了标准 Actor-Critic advantage optimization。更准确的说法是：

```text
We optimize an offline actor-critic surrogate over logged transitions, where the actor is trained by reward-weighted behavior cloning and negative-action avoidance, and the critic fits the logged regret-aware reward.
```

如果后续想更贴近图中公式，需要补充 FQE/CQL/TD target 或者显式 advantage：

```text
A'_t = r_t + gamma * V(s_{t+1}) - V(s_t) + retrospective regret correction
```

当前还没有做到这一步。

## 3. 为什么重新设计并重跑数据切分

旧切分的问题主要有两个。

第一，MDP step 定义不清楚。原始 Yambda 是 timestamped events，没有天然 request/session 边界。之前按 item episode 聚合，容易让一个 item 的很远未来反馈参与当前 reward，解释性弱。

第二，旧 `val/test` 每个用户基本只有 1 条样本。后悔池需要同一用户的时间序列来逐步积累失败路径，如果每个用户只有 1 条，replay 时 pool 还没来得及更新，评估自然会出现：

```text
neg_suppression = 0
pos_suppression = 0
```

这不是 RAPI 没用，而是评估 split 不适合评估后悔池。

所以新切分改成 `session_run`：

```text
原始用户事件流
-> 按时间排序
-> 相邻事件间隔 > 1 hour 时切 session
-> 单个 session 最长限制 6 hours，避免链式超长 session
-> session 内连续同 item 聚合成一个 RL step
-> 每个用户额外保留 replay_val/replay_test，各最多 50 个连续 step
```

当前 runner 是：

```bash
./scripts/run_split_session_run.sh
```

## 4. 一个具体样本从原始事件到训练和评估的全生命周期

假设某个用户在一个 session 内有如下连续事件：

```text
时间顺序:
  A listen played_ratio=0.10
  A like
  A unlike
  B listen played_ratio=0.95
  B like
  C listen played_ratio=0.20
  D dislike
  E listen played_ratio=0.90
```

### 4.1 session-run 聚合

连续同 item 会被聚成一个 RL step：

```text
AAABBBCDE 这种结构会变成:
  A step
  B step
  C step
  D step
  E step
```

上面的例子会变成：

```text
step 1: A = [listen 0.10, like, unlike]
step 2: B = [listen 0.95, like]
step 3: C = [listen 0.20]
step 4: D = [dislike]
step 5: E = [listen 0.90]
```

每个 step 的时间戳取这个连续 run 最后一个事件的时间。例如 A step 的时间戳取 `A unlike` 的时间。

### 4.2 reward 如何计算

当前 v2 reward 大致是：

```text
如果有 listen:
  play_reward = 2 * max_play_ratio - 1
否则:
  play_reward = 0

reward =
  play_reward
  + 0.8 * effective_like
  - 1.2 * effective_dislike
  - 0.6 * effective_unlike
  + 0.2 * effective_undislike

最后 clip 到 [-2, 2]
```

对 A step：

```text
events = [listen 0.10, like, unlike]
max_play_ratio = 0.10
play_reward = 2 * 0.10 - 1 = -0.8
effective_like = 1
effective_unlike = 1

reward_raw = -0.8 + 0.8 - 0.6 = -0.6
reward_scaled = -0.6
regret_type = unlike 或 negative 类
```

对 B step：

```text
events = [listen 0.95, like]
play_reward = 2 * 0.95 - 1 = 0.9
like = 1
reward = 0.9 + 0.8 = 1.7
```

对 C step：

```text
events = [listen 0.20]
play_reward = 2 * 0.20 - 1 = -0.6
reward = -0.6
regret_type = low_play
```

对 D step：

```text
events = [dislike]
play_reward = 0
reward = -1.2
regret_type = dislike
```

### 4.3 训练样本长什么样

训练第 5 个 step `E` 时，数据行大致是：

```text
history_item_ids       = [A, B, C, D]
history_feedbacks      = [-0.6, 1.7, -0.6, -1.2]
history_event_type_ids = [unlike, like, low_play, dislike]
target_dense_item_id   = E
target SID             = dense_item2sid[E]
reward                 = E step 的 reward
```

这里的 history 不再只是 item 序列，而是带上了用户对历史 step 的态度：

```text
item feature + feedback reward + event type
```

这点很重要，因为之前的纯序列推荐只知道用户看过 A/B/C/D，却不知道用户到底喜欢还是讨厌它们。当前 actor 的 state encoder 已经把 `history_feedbacks` 和 `history_event_type_ids` 加进 Transformer 输入了。

### 4.4 训练时如何使用这条样本

actor 输入 history，输出 target item 的 SID path 每一层 token 分布：

```text
P(z1 | history), P(z2 | history, z1), P(z3 | history, z1,z2)
```

如果当前 step 的 reward >= 0：

```text
把 target SID 当作正样本，做行为克隆：
提高 target SID path 的概率
```

如果 reward < 0：

```text
把 target SID 当作负样本，做 avoid loss：
降低 target SID path 的概率
```

critic 同时用 actor 的 context 去拟合这个 logged reward：

```text
q_pred -> reward
```

### 4.5 replay 评估时如何使用这条样本

replay 是按用户时间顺序一条条走，batch_size=1，保证后悔池更新顺序正确。

对于当前 row，先取出该用户已有后悔池：

```text
pool_before = memory.get(user)
```

然后分别算：

```text
base_logp = 不加后悔池时，actor 对当前 target SID 的 log probability
pool_logp = 加后悔池时，actor 对当前 target SID 的 log probability
suppression = base_logp - pool_logp
```

最后根据当前 row 的 reward/regret_type 更新后悔池：

```text
如果当前 step 是 low_play/dislike/unlike 或 reward < 0:
  把当前 target SID path 插入该用户后悔池
  phi 根据 regret_strength 或负 reward 得到
  旧 entry 每次按 gamma=0.9 衰减
```

因此 replay 评估回答的是：

```text
如果用户过去已经对某些语义路径后悔，RAPI 是否会在未来更强地压制类似路径？
```

当前答案是：会，并且在 val/test 上稳定。

## 6. 当前阶段最终效果如何

可以确认的有效结论：

```text
1. 新切分把原始事件变成了更像 RL 的 step：一个 action item 对应一个聚合反馈 reward。
2. history 已经包含历史反馈值和事件类型，不再只是 item 序列。
3. actor/critic 可以在真实 logged transitions 上训练，不再依赖不稳定的 user_response simulator。
4. 后悔池 RAPI 在 replay_val/replay_test 上稳定压制负反馈语义路径，负样本压制约为正样本的 3 倍。
```

不能过度声称的结论：

```text
1. 还不能说最终推荐 reward 提升，因为没有可信 simulator，也没有完成 OPE/FQE。
2. 还不能说 online interaction depth 提升，因为当前没有环境 rollout。
3. actor 的 full SID 命中率仍很低，说明生成完整 item path 的能力还弱。
```

从论文贡献角度，目前最稳的表述是：

```text
The regret memory intervention consistently suppresses future logged negative semantic paths more strongly than positive paths under chronological replay.
```

也就是证明“后悔池干预机制有效”，而不是证明“完整 RL 策略已经在线更优”。

## 7. 当前不足和下一步

### 7.1 不足

1. 当前 loss 不是严格标准 Actor-Critic。没有 TD target，也没有显式 `A'_t` advantage。

2. reward 权重仍是规则设定：

```text
like = +0.8
dislike = -1.2
unlike = -0.6
undislike = +0.2
```

这些权重可以被神经网络学习，但如果直接端到端学，容易重新回到 simulator/reward 混在一起导致不可解释的问题。更稳的做法是小范围可学习、加正负约束和单调约束，例如保证 dislike/unlike 权重为负。

3. replay suppression 不是推荐指标。它只能说明 RAPI 改变了概率分布，不能说明真实用户一定更满意。

4. RAPI 对正样本也有压制：

```text
pos_suppression ~= 0.022 - 0.024
```

说明后悔池不是完全无副作用。后续可以调小 `sara_eta`，或者加 gate：只有当候选路径与高 phi regret path 高度重合时才启用强干预。

5. actor 的 full SID accuracy 很低。token_acc 高于随机，但完整 path 仍很难直接命中，这会影响最终推荐候选质量。

### 7.2 建议下一步

短期应该补两个东西：

```text
1. logged reward/step 统计表：
   session 平均 step 数、reward 均值、负样本比例、按 regret_type 的 reward 分布。

2. RAPI 参数敏感性：
   sara_eta = 0.1 / 0.3 / 0.5 / 0.8
   看 neg_suppression 和 pos_suppression 的 trade-off。
```

中期如果要把图中的 Joint Optimization 做实，需要做：

```text
1. FQE 或 TD critic：
   Q(s,a) <- r + gamma * E[V(s')]

2. 用 advantage 训练 actor：
   A'_t = Q(s,a) - V(s) + retrospective regret correction

3. 或者走 Conservative Offline RL：
   避免 actor 偏离 logged behavior 太远。
```

如果论文短期要推进，建议当前版本先定位为：

```text
Regret-aware offline sequential policy with semantic-path intervention
```

不要强行声称已经完整实现 online RL 或 simulator-based reward optimization。

## 8. 关键问题补充：历史编码、RRCA、loss 对应和 simulator

### 8.1 历史交互物品如何进入 actor

当前 actor 的历史编码流程是：

```text
history_item_ids
-> 通过 dense_item_features.npy 查到每个历史 item 的连续向量
-> item_map 映射到 d_model
-> 加 position embedding
-> 加 history_feedbacks 映射
-> 加 history_event_type_ids embedding
-> TransformerEncoder/SASRec-like causal encoder
-> 取最后一个位置的 state_emb 作为当前用户上下文
-> 逐层生成 target SID token
```

所以回答这个问题：

```text
用户历史交互过的物品会先查到 item feature 向量，再聚合成上下文。
```

但有一个细节：当前不是用历史物品的 SID token 去查向量，而是用 `dense_item_features.npy` 里的 item feature。SID 主要用于两个位置：

```text
1. 训练 actor 时，target_dense_item_id -> dense_item2sid.npy，作为要预测的 SID path label。
2. RAPI 后悔池里存失败 item 的 SID path，用于推理时对相似语义路径做 logits penalty。
```

这和原 HSRL/RL4RS 框架的思路是接近的：原框架也是先把历史 item feature 过 attention/SASRec-like 模块得到 user interest，再对 candidate item 打分。区别是原框架通常直接对 candidate item/slate 打分，而当前策略输出的是层级 SID token。

### 8.2 公式 (6)(7) 的 unlike/undislike 后悔动作是否加了

分两层看。

公式 (6)：

```text
psi_t = -lambda_unlike * I(y_t = unlike)
      +  lambda_undislike * I(y_t = undislike)
```

当前实现里，`unlike` 和 `undislike` 已经进入 reward 规则：

```text
reward =
  play_term
  + 0.8 * like
  - 1.2 * dislike
  - 0.6 * unlike
  + 0.2 * undislike
```

也就是说：

```text
unlike 已经作为负反馈进入 reward，并且会进入 regret_type / negative path。
undislike 当前作为正向修正进入 reward，不进入 negative regret pool。
```

所以公式 (6) 的动作类型有加，但不是完全按论文里的 `psi_t` 单独写成一项，而是被融合进了当前 step 的 reward 与 regret_type。

公式 (7)：

```text
A'_t = A_t + gamma^Delta_t * psi_t
```

当前没有严格实现。当前代码里没有显式 advantage `A_t`，也没有把 unlike/undislike 作为延迟信号回溯到原始决策时间去更新 advantage。

现在近似替代它的是两件事：

```text
1. 负 reward / regret_type 会让该 logged action 在 actor loss 中变成 avoid 样本。
2. replay 时负反馈 SID path 会进入 RegretMemoryPool，未来推理时压制相似语义路径。
```

这能体现“后悔信号影响未来策略”，但还不是“回溯修正历史 advantage”。如果论文里继续保留公式 (7)，需要后续补一个真正的 advantage/TD 版本，或者把文字改成当前实现能支撑的形式。

### 8.3 为什么现在没有加 A 优势函数

核心原因：我们从 simulator-based online RL 暂时切到了 offline logged transition learning。

优势函数需要至少有：

```text
V(s_t)
Q(s_t, a_t)
或 TD target: r_t + gamma * V(s_{t+1})
```

但当前数据只有 logged action 的真实 reward。对于 actor 生成的其他新 item，我们不知道用户会给什么反馈。之前尝试训练 `user_response` simulator，就是想补这个环境，但效果不稳定。

因此当前训练采取更保守的 offline surrogate：

```text
reward >= 0:
  模仿 logged action

reward < 0:
  避开 logged action

critic:
  拟合 logged reward
```

这不是标准 Actor-Critic advantage learning，但在没有可信 simulator 时更稳，不会让策略在不可验证的动作空间里乱跑。

如果要把 `A'_t` 加回来，比较合理的下一步是：

```text
1. 训练一个 V/Q critic 或 FQE critic。
2. 用 logged transition 构造:
   A_t = r_t + gamma * V(s_{t+1}) - V(s_t)
3. 对 unlike/undislike 做 retrospective correction:
   A'_t = A_t + gamma^Delta_t * psi_t
4. actor 用:
   -A'_t * log pi(a_t|s_t)
```

这一步可以做，但它依赖更严谨的 `next_state` 和 value learning，不应该在当前 surrogate 里硬塞。

### 8.4 当前 loss 如何尽量对应图中 Joint Optimization

图中 loss 是：

```text
L_v = E[(V_phi(s_t) - Q_hat(s_t,a_t))^2]
L_pi = E[-A'_t log p_theta(z_l|c_{l-1}) - lambda_en H]
min L_v + L_pi
```

当前实现可对应成一个“离线保守版本”：

```text
critic_loss = MSE(q_pred, reward)

pos_loss = -log pi(logged SID | state), only for positive/reward>=0 samples
neg_loss = -log(1 - pi(logged SID token | state)), for negative/reward<0 samples
entropy = H(pi)

actor_loss = bc_weight * pos_loss
           + avoid_weight * neg_loss
           - entropy_weight * entropy

total_loss = actor_loss + critic_weight * critic_loss
```

可以把它写成和图中更接近的表达：

```text
L_v(phi) = E_D[(Q_phi(s_t,a_t) - r_t)^2]

L_pi(theta) =
  E_D[
    w_t^+ * (-log pi_theta(a_t|s_t))
    + w_t^- * (-log(1 - pi_theta(a_t|s_t)))
    - lambda_en * H(pi_theta)
  ]

w_t^+ = sigmoid((r_t - threshold) / tau)
w_t^- = sigmoid((threshold - r_t) / tau)
```

这个式子比图中原式更符合当前代码。它的含义是：

```text
好反馈的 logged action 要学；
坏反馈的 logged action 要避开；
critic 拟合 logged reward；
entropy 防止策略过早塌缩。
```

不能完全对应原文的原因：

```text
1. 没有 online rollout，所以没有 actor 当前动作的真实反馈。
2. 没有可靠 simulator，所以不能算新动作的 reward/step。
3. 当前 critic 拟合的是 immediate logged reward，不是完整 Q return。
4. negative avoid loss 是 conservative offline RL 风格，原文公式没有这一项。
```

因此可以尝试对应，但建议改写论文公式，而不是硬说代码实现了标准 Actor-Critic。

### 8.5 simulator 是否还需要做

如果最终实验必须给出：

```text
average reward
average interaction step/depth
```

那么确实还需要一个 simulator 或者 OPE。没有 simulator 时，我们只能给 logged replay 指标，不能严谨地说策略 rollout reward。

原 HSRL 的 simulator 是：

```text
policy 推荐 slate items
-> user_response_model 输出每个 item 的 click probability
-> Bernoulli sample 得到 0/1 click
-> reward_func(response) 算 reward
-> 点击 item 加入 history
-> temper 降到阈值则 episode done
```

原框架的 feedback 很简单：只有 click/no-click。Yambda 这里复杂很多：

```text
listen
played_ratio
like
dislike
unlike
undislike
low_play
```

所以不能直接照搬原 HSRL 的 `RL4RSUserResponse`。但可以借它的 env 结构。

建议重新做一个最小可用 `YambdaUserResponseEnv`：

```text
输入:
  当前用户 history step features
  policy 选出的 action item

simulator 输出:
  listen_prob
  play_bucket / played_ratio
  like_prob
  dislike_prob
  unlike_prob
  undislike_prob
  negative_prob / regret_type

reward:
  不让网络直接猜标量 reward
  用预测出的行为事件走 deterministic reward rule

state update:
  把 action item + sampled feedback_signal/event_type 加回 history

done:
  参考原 HSRL temper
  连续负反馈或累计 reward 太低时提前结束
  或达到 max_step_per_episode
```

这比之前旧 `User-response` 更稳的原因：

```text
1. simulator 只学用户行为组件，不直接学最终 reward 标量。
2. reward 保持 rule-based，可解释，和数据切分一致。
3. 训练数据使用新版 session_run transitions，而不是旧 smoke split。
4. 评估 simulator 不再只看 MSE，而看 coarse 行为能力：
   reward sign accuracy
   negative recall / balanced accuracy
   play bucket balanced accuracy
   regret type separation
```

短期最可落地版本：

```text
1. 复用当前 RegretUserResponse 多头模型。
2. 用新版 session_run 数据重新训练，不用旧 smoke 数据。
3. 关闭 learned reward scorer，保持 decoupled rule reward。
4. simulator 评估只要求能区分正负/低播放/显式负反馈，不要求精确回归 reward。
5. 接入一个 env wrapper，输出 reward 和 done。
```

如果它能在 held-out 上做到：

```text
negative balanced accuracy > 0.60
reward sign accuracy > 0.65
play bucket balanced accuracy 明显高于随机
并且 by_type 里 none 的预测 reward 明显高于 low_play/dislike/unlike
```

就可以作为“可用 simulator”去跑 reward/step 指标。否则 reward/step 指标会非常虚，容易被审稿人抓住。

### 8.6 下一步建议

下一步不要再回到旧的混乱 `User-response` 实验方式，而是按下面顺序做：


这条路和当前 RAPI/offline actor 不冲突。当前 RAPI 证明“后悔池干预方向有效”；simulator 是为了补“策略 rollout reward/step”。

### 10.4 如何解释这个结果

这个结果是“方向正确但幅度较小”：

```text
1. RAPI 让平均累计 reward 提升约 +1.74%。
2. RAPI 让每步 reward 提升约 +0.0055。
3. RAPI 让负反馈率下降约 0.29 个百分点。
4. RAPI 对平均交互步数几乎没有影响。
```

可以说它支持一个弱结论：在当前 simulator 下，后悔池 soft mask 会轻微降低负反馈并轻微提升累计 reward。但不能把这个结果当作强主实验，因为 simulator 本身还不够可靠。

### 10.5 造成当前结果幅度有限的原因

第一，当前 simulator 是弱的。它能给出 reward/step 接口，但行为预测仍然偏多数类：listen 几乎总为 1，play 偏 high，undislike 概率异常高，negative balanced accuracy 只在 0.54 左右。因此 rollout 的绝对 reward 和 step 会被 simulator 偏差主导。

第二，actor 本身不是严格按 Eq.11-Eq.13 的在线 Actor-Critic 训练出来的。当前 actor 更接近 offline behavior cloning + negative avoid + critic MSE 的保守离线代理目标，所以它首先学的是“真实日志里常出现的 SID 路径”，不是直接最大化 simulator reward。

第三，RAPI 是 inference-time soft intervention，不是重新学习策略。它只在候选 SID path 与 regret pool 里的失败语义路径发生重叠时改变 logits；如果 top-k 候选本身没有命中后悔池，或者 actor logits 很集中，RAPI 的可作用空间就有限。

第四，rollout 设置了 `max_steps=20`，而 base 和 RAPI 的 early stop rate 都很低，所以平均 step 接近上限。此时 step 指标天然不敏感，RAPI 更容易体现在 reward_per_step 和 negative_rate，而不是总 step。

第五，后悔池目前只使用语义路径相似度做 soft mask，没有区分更细的 regret 来源。也就是说 low play、dislike、unlike 都会进入类似的路径压制逻辑，尚未真正实现“外部/内部 regret 分流”或不同类型 regret 的差异化策略。

### 10.6 仍然遗憾没完全实现的地方

严格的优势修正还没有实现。论文 Eq.6-Eq.7 里的 `psi_t` 和 `A'_t = A_t + gamma^dt psi_t` 目前只被近似成 reward 规则、negative avoid loss 和 RAPI regret memory，还没有形成完整的 TD advantage 或 FQE advantage 管线。

严格的 Actor-Critic 三个 loss 也没有完全对应原文。当前 critic 是 `Q(s,a)` 对 logged reward 的 MSE，actor 是 token-level BC/avoid surrogate，并没有稳定估计 `V(s)`、`Q(s,a)`、`A_t` 后再用 `A'_t log pi(a|s)` 优化。

simulator 还没有达到可以作为强评估器的质量。它目前适合做相对 sanity check，不适合作为主结果证明模型真的提升了用户满意度。主证据仍应优先使用 replay 上的 negative suppression / positive suppression 差异。

RAPI 还没有被纳入训练闭环。现在是“训练一个 actor，然后推理时加后悔池”。更强的版本应该是训练时就把 regret-aware logits 或 regret-augmented advantage 纳入 actor loss。

reward 权重仍是人工规则。虽然可以让 `omega_listen / omega_like / omega_dislike / lambda_unlike / lambda_undislike` 变成可学习参数，但需要约束符号和范围，否则会破坏可解释性。当前阶段为了稳定，仍采用固定规则。

### 10.7 当前阶段可写进论文/报告的结论

稳妥表述：

```text
On replay evaluation, the regret-aware policy intervention suppresses future negative semantic paths more strongly than positive paths, indicating that the regret memory acts as a targeted semantic-level intervention.
```

辅助表述：

```text
In a learned simulator rollout, RAPI obtains a small positive reward gain (+0.106 cumulative reward over 20-step rollouts) and a slight reduction in negative feedback rate, while maintaining nearly unchanged interaction depth.
```

不建议强写：

```text
RAPI significantly improves true user reward.
RAPI clearly increases interaction depth.
The simulator accurately models user preference dynamics.
```

## 11. Advantage/RRCA 接入实验：adv_v1 到 adv_v1_1_warm

### 11.1 实验目的

前面已经确认：当前 `regret_sid_session_run_v2` 不是严格 Actor-Critic，而是离线监督 surrogate：

```text
正 reward 样本：提高 logged SID path 概率
负 reward 样本：降低 logged SID path 概率
critic：拟合 logged immediate reward
没有显式 V(s)
没有 A_t = Q(s,a) - V(s)
没有 A'_t = A_t + gamma^dt psi_t
```

本阶段目标是补齐 Eq.6-Eq.7/Eq.11-Eq.13 的工程链路：

```text
1. 增加 V(s) baseline。
2. 增加 Q(s,a) critic。
3. 计算 A = Q - V。
4. 加入 RRCA correction。
5. 用 corrected advantage 影响 actor loss。
```

### 11.2 实现版本

#### adv_v1

实现内容：

```text
新增 StateValueCritic。
使用旧 TokenRewardCritic 输出 q_pred。
advantage = q_pred - v_pred。
corrected_advantage = advantage + rrca_correction。
actor loss = old surrogate + adv_loss_weight * advantage_policy_loss。
```

问题：

```text
TokenRewardCritic 没有显式输入 logged action。
它更像 Q(s, pi(s)) 或另一个 state value。
因此验证集上 Q-V 容易退化为常数，advantage 没有足够区分度。
```

结果：

```text
adv_v1 val_loss = 20.7108
adv_v1 val_q_mse = 0.6978
adv_v1 val_v_mse = 0.7008
adv_v1 val_token_acc = 0.0427
```

结论：链路能跑，但 Q 设计不对，不能作为正式 after。

#### adv_v1_1

修正内容：

```text
新增 SIDActionQCritic。
Q(s,a_logged) 显式输入 state_emb + target SID token embeddings。
V(s) 只输入 state/context。
advantage = Q(s,a_logged) - V(s)。
```

默认从零训练结果：

```text
adv_v1_1 val_loss = 20.6979
adv_v1_1 val_q_mse = 0.6938
adv_v1_1 val_v_mse = 0.7054
adv_v1_1 val_token_acc = 0.0427
corrected_adv_pos_share ≈ 0.0001
```

结论：Q(s,a) 的 reward 拟合更好，但 actor 被 advantage 项拖坏，不能作为正式 after。

#### adv_v1_1_warm

稳定版配置：

```text
actor_init_checkpoint = artifacts/models/regret_sid_session_run_v2_actor
q_critic_type = sid_action
adv_loss_weight = 0.01
rrca_weight = 0.1
critic_lr = 0.0003
value_lr = 0.0003
max_train_rows = 1000000
epochs = 3
```

训练结果：

```text
adv_v1_1_warm val_loss = 20.5484
adv_v1_1_warm val_q_mse = 0.6941
adv_v1_1_warm val_v_mse = 0.7020
adv_v1_1_warm val_token_acc = 0.0427
corrected_adv_pos_share = 0.1369
```

对比 before：

```text
baseline v2 val_loss = 20.2383
baseline v2 val_q_mse = 0.7002
baseline v2 val_token_acc = 0.0519
```

训练侧结论：

```text
Q(s,a) 拟合 reward 有改善。
V(s) baseline 可稳定训练。
advantage/RRCA 链路不再完全塌缩。
但是 actor token accuracy 和 val_loss 仍弱于 baseline。
```

### 11.4 阶段结论

这一阶段不是“正式主训练胜利”，但完成了一个重要机制补齐：

```text
1. Q(s,a_logged) 已经显式 action-conditioned。
2. V(s) baseline 已经接入。
3. A=Q-V 和 RRCA correction 已经能被记录和进入 actor loss。
4. replay_val full 显示 regret-targeted suppression 有小幅提升。
```

同时存在明确不足：

```text
1. actor token accuracy 低于 baseline。
2. val_loss 低于 adv_v1/adv_v1_1，但仍高于 baseline。
3. corrected_advantage 仍整体偏负，说明 advantage policy loss 还不能作为主导训练目标。
4. 当前只跑了 replay_val full，replay_test full 尚未补齐。
```

因此当前最稳妥表述是：

```text
The advantage/RRCA-enhanced actor does not improve behavior-cloning token accuracy, but it yields a small and consistent improvement in regret-targeted replay suppression on validation replay.
```

### 11.5 下一步方向

不继续盲目调 advantage actor loss。下一阶段应转向：

```text
1. 补全 replay_test full，确认 replay_val 的提升能否泛化。
2. 增加分类型 replay suppression：low_play / dislike / unlike 分开报告。
3. 将 RAPI 评估结果与 advantage/RRCA 训练结果解耦报告。
4. 保留 adv_v1_1_warm 作为机制增强 ablation，而不是替代 baseline 主模型。
5. 后续如果继续优化 actor loss，应采用更保守方案：只用 corrected advantage 做 sample reweighting，而不是直接做负优势 policy gradient。
```
