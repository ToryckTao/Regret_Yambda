## -3. 2026-05-10 P0 对齐记录：RRCA 改为 effective reward

本轮目标：对齐最新论文 2.3 和 2.5，不再默认把 retrospective revision 只加到 actor advantage 上，而是先重构有效奖励，再由同一个 `r_eff` 同时驱动 critic target 和 actor advantage。

已改文件：

```text
scripts/10_train_hsac_simulator_rollout.py
backups/2026-05-10_effreward/10_train_hsac_simulator_rollout.py.bak
```

核心逻辑：

```text
raw_reward      = simulator 原始输出，仍用于环境历史追加，保证 simulator 输入分布不突然漂移
base_reward     = actor-critic 使用的基础奖励
reward_correction = gamma^Delta * psi
effective_reward  = clamp(base_reward + reward_correction)
target          = effective_reward + gamma * V_target(s')
A               = target - V(s)
actor_loss      = -A * log pi - entropy
critic_loss     = mse(V(s), target)
```

默认参数已改为论文对齐版本：

```text
--base_reward_mode no_revision_v2
  使用当前 v2 的 play/like/dislike 尺度，但不把 unlike/undislike 直接计入 base reward。

--rrca_apply_to effective_reward
  RRCA 默认重构 reward，而不是只修正 advantage。

--rrca_signal_scope revision_only
  psi 默认只来自 unlike / undislike：
  unlike    -> -lambda_unlike
  undislike -> +lambda_undislike
```

保留消融开关：

```text
--rrca_apply_to advantage
  回到旧版 A-only correction。

--rrca_apply_to off
  关闭 RRCA correction。

--rrca_signal_scope all_negative
  回到旧版 low_play / dislike / unlike / undislike 都参与 correction 的策略。

--base_reward_mode simulator
  直接使用 simulator 原始 reward，保留旧行为用于对照。

--base_reward_mode paper
  使用论文公式字面版本：w_listen * played_ratio + w_like * like - w_dislike * dislike。
```

### -4.2 预计算 per-user cross-session B_rev snapshot

已改文件：

```text
scripts/02_split_transitions.py
scripts/run_split_session_run.sh
regret_core/data/transition_dataset.py
regret_core/model/regret_memory.py
```

设计：

```text
切分阶段按 user 时间顺序扫描所有 session-run steps。
对每个 step，在写出当前 transition row 之前，先取该用户此前历史中的 failed memory snapshot。
snapshot 写入三列：
  regret_memory_item_ids
  regret_memory_phis
  regret_memory_type_ids
然后再根据当前 step 是否 low_play/dislike/unlike 更新该用户 memory。
```

这样训练时即使 dataloader 随机读 row，也不会造成 per-user memory 时间乱序。当前旧数据没有这些列时，训练脚本会自动 fallback 到 history 初始化；新数据有这些列时，优先使用预计算 snapshot。

```text

--regret_memory_size 20
--regret_memory_gamma 0.9
--regret_memory_min_phi 1e-6
--regret_memory_scope all_failed
```

## -2. 最新论文版本与当前实现的不一致清单

本节记录 2026-05-10 对照最新论文 2.1-2.5 后发现的全部关键不一致，后续实现必须逐项消解或在论文中明确解释。

### -2.1 2.1 Problem Formulation: action 是 k-item list，但当前实现是单 item

论文当前写法：

```text
An action a_t is a list of k items.
```

当前代码实现：

```text
每个 env.step 只生成一个 semantic ID。
SID decoder 将该 semantic ID 映射成一个 dense item。
simulator 对这个单 item 生成一次 composite feedback 和 reward。
```

差距：

```text
代码当前相当于 k=1。
没有实现 slate/list recommendation。
没有实现一轮推荐多个 item 后的排序加权 reward。
```

修改选择：

```text
优先低成本方案：论文中明确说明当前实现取 k=1。
高成本方案：扩展 actor/decoder/env/simulator，使一次 action 生成 k 个 item，并定义 slate reward。
当前不建议立刻做高成本方案。
```

### -2.2 2.1 Step 定义与当前实现基本一致，但需要论文补充 session boundary

论文当前写法：

```text
time step = timestamp interval between two consecutive changes in the target item.
```

当前代码实现：

```text
先按 1 hour gap 拆 session。
再在每个 session 内把相邻同 item events 聚合成 step。
step_time = step_end_time = run 内最后一个 event timestamp。
```

差距：

### -1.2 Δtr 和 revision memory

论文公式 8 的广义时间差定义：

```text
Δtr = revision signal 所在 step 与原始 action step 之间的 count-based step distance。
这里的 step 是上面定义的 session-run 聚合 step，不是原始 event。
Δtr 可以跨 session，但必须在同一用户历史内计算。
```

revision memory 定义：

```text
B_rev 永远是 per-user memory。
它应该维护用户全历史里的 failed semantic paths，而不是只维护当前 session。
因此 B_rev 允许跨 session 延续。
```

当前代码状态：

```text
当前 RegretMemoryPool 是 batch/rollout 内局部对象。
训练和评估时会从当前 state 的 history 初始化一部分 memory，并在 rollout 内继续更新。
它还不是严格意义上的 per-user persistent cross-session memory。
```

需要改造：

```text
方案 A：在 transition 构造阶段为每个 row 预计算截至当前 step 前的 B_rev 摘要。
方案 B：训练/评估时按 user 时间顺序流式处理，维护全局 user_id -> B_rev。

如果 loader 继续 shuffle，则方案 B 会破坏时间顺序。
因此更稳妥的是先做方案 A，或者在需要 persistent memory 的训练/评估中关闭 shuffle 并按 user/time 排序。
```

预计算 B_rev 的含义：

```text
不是把未来推荐写死。
而是在每个真实训练样本 row 上附带“该用户在当前 step 之前已经形成的长期 revision memory 快照”。
训练时即使随机读取 row，也不会依赖 loader 顺序。
```

新推荐 item 的处理：

```text
预计算 B_rev 只提供 rollout 起点之前的历史记忆。
训练/评估过程中，actor 新推荐 item 后，如果 simulator 返回 negative/revision feedback，
仍然会在线追加进当前 episode 的临时 B_rev，并影响后续 rollout step。
因此长期历史 memory 和当前 rollout 临时 memory 是两层：
1. precomputed user-level B_rev snapshot: 截至初始 state 前的跨 session 历史。
2. online episode B_rev updates: rollout 中新推荐产生的即时 revision memory。
```

当前暂不实现 precomputed B_rev；先完成 A' 当前 step 和统计验证。

```text
undislike 按公式 6 参与 A' 的正向 callback：ψ_t = +λ_undislike。
undislike 不进入 B_regret，因为它不是 failed semantic path。
扩大训练前需要看 callback 类型拆分，而不能只看 callback_per_step。
```

## 0.2 RRCA 符号核对记录

公式约定：

```text
ψ_t = -λ_unlike * I(y_t = unlike) + λ_undislike * I(y_t = undislike)
```

当前代码实现：

```text
dislike   -> psi_signed < 0, phi_penalty > 0, regret_type_id = 2, 写入 B_regret
unlike    -> psi_signed < 0, phi_penalty > 0, regret_type_id = 3, 写入 B_regret
low_play  -> psi_signed < 0, phi_penalty > 0, regret_type_id = 1, 写入 B_regret
undislike -> psi_signed > 0, phi_penalty = 0, regret_type_id = 0, 不写入 B_regret
```

轻量代码验证：

```text

unlike    = (-0.6, 0.6, 3)
undislike = (0.2, 0.0, 0)
low_play  = (-0.7, 0.7, 1)
```

真实 transition shard 抽样验证：

```text
rows = 50000
effective_dislike:   n=95,  mean_reward_raw=-1.3979
effective_unlike:    n=104, mean_reward_raw=-0.5635
effective_undislike: n=9,   mean_reward_raw=0.1244
```

解释：

```text
undislike 的单项权重是正的。
但如果同一个 step 中同时存在有效 dislike 和 undislike，整条 step reward 仍可能为负。
因此 undislike 应作为 A' 的正向 correction，但不应作为 failed semantic path 放入 B_regret。
```

## 1. 当前方向修正

前一阶段把问题偏向了 offline RL / logged transition surrogate：

```text
state -> logged item -> logged reward
用离线 transition 学 actor/critic
RAPI 在 replay 上做 suppression 评估
```

这个方向能跑通 replay，但和论文想表达的 HSAC/RRCA/RAPI 闭环仍有距离。

现在方向应调整为：尽量靠近 `/root/autodl-tmp/HSRL_Original` 的训练范式。

核心变化：

```text
我们不再把 user_response/simulator 完全排除在主流程之外。
用户反馈由 simulator 给出，虽然 simulator 质量有限，但它可以提供在线 rollout 中的 reward、step、regret event。
RRCA 的后悔回调应该发生在训练/rollout 过程中，而不是只在静态 parquet 里预先写死。
RAPI 后悔池是随 rollout 动态更新的机制，不应被简化成离线 replay suppression。
```

## 2. 用户对“后悔回调”的定义

用户明确指出：后悔回调不应只是离线数据构造阶段对历史样本写 correction。

更符合论文的定义是：

```text
1. 当前策略在 rollout 中预测一个新 item/action。
2. simulator 返回用户反馈。
3. 如果反馈是 regret action，例如 unlike / undislike / dislike / low_play，则系统识别为后悔信号。
4. 该后悔信号会回调到对应的原始决策时间 t_orig，修正当时的 advantage：
   A'_t = A_t + gamma^delta_t * psi_t
5. 同时该失败语义路径会进入 B_regret 后悔池。
6. 后悔池影响后续 action token 生成，即 RAPI 的公式 8/9/10。
```

也就是说，RRCA 和 RAPI 在训练/rollout 中是连续发生的：

```text
policy action -> simulator feedback -> RRCA advantage correction -> update regret pool -> next policy action with RAPI soft mask
```

补充澄清：

```text
公式 6 的符号是 psi，写作 ψ_t，表示 regret intensity。
公式 8 的 Φ_j = γ^Δt · ψ_t。
因此 ψ_t 不只用于公式 7 的 A'_t，也应该作为后悔池条目的强度来源。
```

对应到实现：

```text
同一个 regret signal ψ_t 应该同时进入两条路径：
1. RRCA: A'_t = A_t + γ^Δt · ψ_t
2. RAPI: B_regret 插入 (z_{1:L}, Φ)，其中 Φ = γ^Δt · ψ_t
```

## 3. 时间步问题

时间步不应只理解成 parquet 行号。

在 HSRL/simulator rollout 中，时间步应是环境交互 step：

```text
t = rollout 内第几个推荐/交互 step
delta_t = regret event step - original decision step
```

当用户对当前推荐 item 给出 regret action 时：

```text
如果该 regret 是针对当前 action 立即发生，delta_t = 0。
如果是对历史推荐 item 的 unlike/undislike，则需要找到该 item 最近一次被推荐/正反馈的 step 作为 t_orig。
```

这里需要进一步确认数据和 simulator 是否能区分：

```text
unlike 是针对当前推荐 item？
还是针对用户历史里某个已 like 的 item？
```

如果 simulator 只能对当前 action 返回 feedback，则第一版只能实现 immediate regret callback，即 `delta_t=0`。

当前理解：

```text
simulator 输入当前历史编码和当前推荐 item，输出的是用户对当前推荐 item 的反馈。
因此第一版默认 regret feedback 作用于当前推荐 item。
```

需要注意：

```text
如果当前 item 在历史中曾经被 like，而当前反馈是 unlike，
那么语义上仍然存在“回调到历史正反馈/原始推荐 step”的需求。
但如果 simulator 不返回明确的 target historical step，第一版先采用 rollout 内可追踪的最近同 item step；
找不到历史 step 时退化为 current-step callback。
```

## 4. MLC：标准多层价值函数

当前 `adv_v1_1` 的 critic 是：

```text
Q(s,a_logged) = f(state_emb, target_sid_embeddings)
V(s) = g(state_emb)
```

这不是论文图里的 MLC。

标准 MLC 应该更接近 HSRL_Original：

```text
HPN 生成每层 token 时都有 context c_l。
MLC 对每层 context 输出 V_phi(s,l)。
通过可学习 layer weights 聚合：
  w_l = softmax(raw_w_l)
  V_hat(s) = sum_l w_l * V_phi(s,l)
```

需要实现：

```text
1. critic 输出 v_seq: (B, L+1)
2. token_weight: learnable softmax weights
3. V_hat = weighted sum(v_seq)
4. 日志记录各层 value 和 layer weight
```

## 5. Loss 聚合

应尽量贴近论文 Eq.11-13：

```text
critic_loss = MSE(V_hat(s_t), Q_hat(s_t,a_t))
actor_loss = - A'_t * log pi(a_t|s_t) - entropy
total_loss = critic_loss + actor_loss
```

但在 simulator 闭环中，`Q_hat` 不应只来自 logged reward，而应来自 rollout：

```text
Q_hat(s_t,a_t) = r_t + gamma * V_target(s_{t+1})
A_t = Q_hat(s_t,a_t) - V_hat(s_t)
A'_t = A_t + gamma^delta_t * psi_t
```

如果为了稳定，可以先用 AWR 风格替代直接负优势 policy gradient：

```text
actor_loss = exp(clamp(A'_t)) * NLL(action)
```

但这是否偏离论文，需要再判断。

## 6. RAPI 解耦

RAPI 不需要参数网络化。

当前 RAPI 更适合作为 inference/rollout-time 机制：

```text
B_regret stores failed SID paths and phi.
D_{z_l=k} computes overlap penalty.
logits_l = logits_l - eta * D_{z_l=k}
```

RAPI 可以继续解耦评估：

```text
base rollout: actor without RAPI
RAPI rollout: same actor with dynamic regret pool
```
