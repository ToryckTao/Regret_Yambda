# 0415IDEA: 下一版 Yambda-HSRL 改造思路

## 当前目标

现有 `0408Yambda` 已经可以作为 HSRL 在 Yambda 上的 baseline：能完成 SID 构建、数据切分、HPN warmstart、UserResponse 模拟器、DDPG 微调和候选集排序评估。接下来不应该继续只修 baseline，而应新开一个实验文件夹，在总体 HSRL 思路不变的前提下加入我们的改动。

建议新文件夹暂定为：

```text
0415Yambda/
```

核心创新方向分三块：

1. 语义 ID 编码改造：从 raw embedding RQKMeans / RQVAE 对比，转向 `PCA 压缩 + RQKMeans`。
2. 数据预处理改造：围绕单用户单 item 的交互轨迹构造 reward，但要保留时间顺序和 RL transition 语义。
3. 反悔池改造：基于用户 regret 记忆，对 HPN 的语义路径生成做 soft intervention。

---

## 方向一：语义 ID 编码改造

### 当前问题判断

现有 baseline 遇到的问题可以拆成三类：

1. **原 HSRL 的 RQKMeans 直接作用在音乐 embedding 上可能不稳定。**
   音乐 embedding 本身是连续高维预编码，如果直接做 RQKMeans，可能出现 full SID 碰撞高、语义局部性不足、层级结构不清晰等问题。是否“利用率低”需要重新验证；我本地对 Stage 01 备选 RQKMeans codebook 做 100k 抽样时，利用率和熵其实不错，但这不是最终结论，需要云端全量诊断确认。

2. **RQVAE 能改善利用率和碰撞，但不天然保证层级语义局部性。**
   RQVAE 优化目标更偏重重构误差、commit loss、codebook 使用等。即使 reconstruction MSE 很低，也只说明重构向量接近原 embedding；不等价于“相似 item 共享 SID 前缀”。如果 RQVAE 没有显式约束前缀层级语义，那么两个 embedding 近邻可能拥有完全不同的 SID 前缀。

3. **我们需要在重构质量和语义前缀结构之间找平衡。**
   推荐中的 SID 不只是压缩编码，还承担 action space 分解功能。理想 SID 应该同时满足：
   - 每层码字充分使用。
   - full path 碰撞低。
   - 重构误差低。
   - embedding 近邻在 SID 空间也尽量近，尤其是共享前缀。
   - 下游 actor 能更容易学习。

### 新思路：PCA 压缩后再 RQKMeans

我们希望先对原始音乐 embedding 做 PCA 压缩，再在压缩空间上做 RQKMeans。

动机：

- PCA 可以先去掉噪声维度和冗余相关性，让 KMeans 面对更规整的空间。
- 降维后 RQKMeans 的聚类可能更稳定，减少高维空间距离退化问题。
- 如果 PCA 保留主要方差信息，RQKMeans 仍然能保留大部分语义相似性。
- 比 RQVAE 更简单、更可解释，也更接近原 HSRL 的 semantic ID 构建路线。

建议对比组：

| 编码方案 | 作用 | 需要比较 |
|---|---|---|
| Raw RQKMeans | 原 HSRL 近似方案 | baseline |
| RQVAE | 当前 0408 最终实际使用方案 | 对照强基线 |
| PCA-32 + RQKMeans | 强压缩方案 | 看碰撞/局部性/重构损失 |
| PCA-64 + RQKMeans | 中等压缩方案 | 重点候选 |
| PCA-96 + RQKMeans | 轻压缩方案 | 看是否更保留原 embedding |
| PCA-whiten + RQKMeans | 可选 | 看 whiten 是否改善码字均衡 |

### 需要计算的指标

#### 1. 码本利用率

每层 256 个 token 中有多少被使用。

```text
used_codes_l / 256
```

只看是否用满不够，还要看分布是否偏斜。

#### 2. 每层 token 熵

衡量 token 分布是否被少数 token 主导。

```text
H_l = - sum p(z_l) log2 p(z_l)
entropy_ratio = H_l / log2(256)
```

`entropy_ratio` 越接近 1，说明越均匀。注意：过于均匀也不一定代表语义好，只说明没有明显塌缩。

#### 3. 每层分布偏斜

建议同时看：

- max bucket share
- min nonzero bucket count
- Gini coefficient
- top-10 token share

这些比“是否用满 256 个码字”更敏感。

#### 4. full SID 碰撞率

多个 item 是否共享完全一样的 4 层 SID。

```text
unique_paths
collision_groups
collision_items_excess = sum(count(path)-1 for count > 1)
collision_rate = collision_items_excess / n_items
```

碰撞不是绝对不能有，但碰撞高会让多个 item 在 semantic action space 里不可区分。

#### 5. prefix 碰撞率

分别看前 1/2/3/4 层 prefix 有多少 unique prefix 和最大 bucket。

这个指标很重要，因为 HSRL 是 coarse-to-fine 生成。如果第 1 层或第 2 层过于拥挤，actor 的粗粒度决策会很难。

#### 6. 重构 MSE / cosine error

这里的 MSE 要说清楚：

- Raw RQKMeans：用 codebook residual sum 重构原 embedding。
- PCA+RQKMeans：可以在 PCA 空间算 MSE，也可以 inverse PCA 回原空间算 MSE。
- RQVAE：用 decoder/reconstruction 算原 embedding MSE。

建议同时报告：

```text
MSE(original_embedding, reconstructed_embedding)
1 - cosine(original_embedding, reconstructed_embedding)
```

MSE 低是好事，但它不等价于前缀共享好。

#### 7. embedding 近邻的 SID 局部性

这比 MSE 更贴近“相似物品是否共享语义前缀”。

做法：

1. 抽样 N 个 item。
2. 在原 embedding 空间找每个 item 的 top-K nearest neighbors。
3. 统计这些 nearest neighbors 和原 item 的 SID 关系。

指标：

```text
mean SID Hamming distance
same L1 prefix ratio
same L1-L2 prefix ratio
same L1-L2-L3 prefix ratio
same full SID ratio
mean common prefix length
```

如果 MSE 低但 `same L1-L2 prefix ratio` 很低，说明重构好但层级语义不适合 HSRL。

#### 8. prefix bucket 的近邻召回

这是更强的局部性指标。

对每个 item 的 embedding top-K 近邻，检查有多少落在同一个 SID prefix bucket 中：

```text
prefix_recall@K(depth=1/2/3)
```

例如 depth=2 时，如果同前两层 SID 的 bucket 能覆盖较多 embedding 近邻，说明 SID prefix 有检索意义。

#### 9. 下游指标

最终不能只看编码指标，还要看下游：

- HPN/SASRec warmstart token acc。
- candidate ranking 的 HR/NDCG/MRR。
- RL rollout total reward。
- regret-aware 版本的 regret event 减少情况。

### 初步评价

`PCA + RQKMeans` 是合理创新点，优点是：

- 和原 HSRL 技术路线更一致。
- 比 RQVAE 更容易解释。
- 可以通过 PCA 维度控制压缩程度。
- 指标可控，容易做消融。

风险是：

- PCA 降维可能丢掉细粒度音乐偏好。
- 降维后 collision 可能更高。
- 如果 PCA 只保留全局方差，不一定保留推荐相关的局部结构。
- 如果 Yambda embedding 已经是模型训练出的语义空间，PCA 未必比原空间好。

需要实验决定，不能先验认定 PCA 一定优。

### 需要和你确认

1. PCA 维度优先跑哪些？我建议先跑 `32/64/96`。
2. PCA 是否 whiten？我建议 `no-whiten` 和 `whiten` 都小样本对比。
3. SID 仍保持 `4 层 × 256` 吗？为了和 baseline 可比，我建议先保持不变。
4. MSE 是以原始 normalized embedding 为目标，还是以 PCA 空间为目标？我建议两者都报，但主指标用原空间 cosine error 和 prefix locality。

## 方向二：数据预处理与 split 改造

### 你的想法

你现在想保留“单用户单 item”的思路：针对每一个用户对每一个 item 的交互历史，聚合出一个 reward。

例子：

```text
[i1 listen, i2 dislike, i3 like, i1 like]
```

直觉上会形成三条数据：

```text
i1: listen + like -> reward_i1
i2: dislike       -> reward_i2
i3: like          -> reward_i3
```

这个方向是合理的，因为 Yambda 的 multi-event 本来就不是“每次推荐只对应一个单一反馈”，同一个 item 可能有多个后续动作。把 user-item 交互聚合成一个 reward，可以更好表达用户对这个 item 的最终态度。

### 必须注意的问题

关键风险是：**不能为了聚合同一个 item 而破坏用户时间顺序。**

如果把某个用户全历史里所有 i1 事件都抽出来合并，那么它会丢失这些事件中间穿插的 i2、i3、i4。对推荐/RL 来说，用户在 i1 之后马上去交互 i2，这本身就是状态转移的一部分，不能随意删除。

所以更稳妥的定义不是“全局 user-item 聚合”，而是：

> 按用户时间线构造 user-item episode。每个 episode 是用户在某个时间窗口内对同一个 item 的完整交互片段。episode 有 start_time、end_time、history_before、reward、next_history_after。

这样能同时保留：

- 单用户单 item 的反馈聚合。
- 用户全局时间顺序。
- RL 需要的 state-action-reward-next_state。

### 推荐的数据记录格式

建议新 split 输出不再只叫 `train.tsv`, 而是明确叫 transition 数据：

```text
train_transitions.tsv
val_transitions.tsv
test_transitions.tsv
```

每行建议字段：

| 字段 | 说明 |
|---|---|
| `transition_id` | 唯一行 id |
| `user_id` | 用户 id |
| `episode_start_time` | user-item episode 开始时间 |
| `episode_end_time` | user-item episode 结束时间 |
| `target_dense_item_id` | 当前被交互的 item，也可视为 logged action |
| `target_orig_item_id` | 原始 item id |
| `event_types` | episode 内事件类型列表 |
| `play_ratios` | episode 内 listen 的播放比例 |
| `history_item_ids` | episode 开始前的用户历史 item 序列 |
| `history_feedbacks` | 对应历史反馈值 |
| `history_event_types` | 可选，保留历史事件类型 |
| `reward_raw` | 原始聚合 reward |
| `reward_scaled` | 缩放或裁剪后的 reward |
| `next_history_item_ids` | episode 结束后的 next state |
| `next_history_feedbacks` | next state 的反馈序列 |
| `has_like` | episode 内是否 like |
| `has_dislike` | episode 内是否 dislike |
| `has_unlike` | episode 内是否 unlike |
| `has_undislike` | episode 内是否 undislike |
| `regret_type` | none / unlike / undislike / low_play |
| `regret_strength` | regret 强度 |
| `original_like_time` | 如果 unlike，追溯到原 like 时间 |
| `split` | train / val / test |

### reward 怎么算

当前可先用固定权重，作为第一版：

```text
reward_raw =
  w_play    * max_play_ratio
  + w_like  * I(like)
  - w_dislike * I(dislike)
  - w_unlike  * I(unlike)
  + w_undislike * I(undislike)
```

可以暂定：

```text
w_play = 1.0
w_like = 1.0
w_dislike = 1.0
w_unlike = 1.0 或 1.5
w_undislike = 0.5 或 1.0
```

但我建议不要马上把这个当最终设计。需要检查 reward 分布后再定。

### reward 可不可以是负数

可以是负数。RL 里负 reward 完全合理，尤其 dislike/unlike 本来就应该惩罚。

但要注意当前环境实现里可能隐含 reward 范围假设。例如 `temper` 终止逻辑如果用：

```text
temper_down = (-reward + 1) + 1
```

负 reward 会让 episode 更快结束。这可以是合理设计，也可能导致训练不稳。因此建议保留两套值：

```text
reward_raw
reward_scaled
```

`reward_raw` 用于分析，`reward_scaled` 用于模型训练。训练版可以考虑：

```text
reward_scaled = clip(reward_raw, -1, 2)
```

或者标准化：

```text
reward_scaled = (reward_raw - mean_train_reward) / std_train_reward
```

具体选哪个需要看 reward 分布后再定。

### 为什么会有那么多条数据

当前报告里 `train=41,513,562`, `kept_events=23,100,324`，按“一条 episode 生成一条样本”的逻辑不应该出现 `train rows > kept events`。这个必须复查，可能是：

- resume/追加写入导致重复。
- split 脚本统计口径错误。
- train.tsv 是旧 run 累积产物。
- 代码实际生成了多于 episode 的样本。

但即使修正后，一个用户产生上千条甚至上万条 transition 也是可能的。原因是：

- Yambda 用户活跃度长尾很强。
- 一个重度用户可能有大量 listen。
- 如果按 user-item episode 切，用户交互 item 多时会产生大量 episode。

因此必须输出这些诊断：

```text
rows_per_user: mean / p50 / p90 / p99 / max
events_per_user: mean / p50 / p90 / p99 / max
episodes_per_user: mean / p50 / p90 / p99 / max
rows_per_user vs events_per_user
```

### 能不能训练时流式读取

可以，而且应该做。

当前 `04_train_hpn_warmstart.py` 和 `05_train_user_response.py` 都偏向把 TSV 读进 pandas，这对 4100 万行级别不适合。新版本建议：

- split 输出 Parquet 或 sharded TSV。
- PyTorch 训练用 `IterableDataset`。
- 按 shard streaming 读取。
- feature cache 不要一次性为全量 item 建 dict。
- 对 item embedding 用 mmap / Arrow / LMDB / numpy memmap。

推荐文件格式：

```text
artifacts/processed/train_transitions/part-00000.parquet
artifacts/processed/train_transitions/part-00001.parquet
...
```

### 这些数据应该在哪里用

你说“上述划分好的数据应该是在训练阶段也就是 06 用”，这个判断对 RL 主线是对的。

更准确地说：

- 新版 `train_transitions` 应该是 RL 训练的核心数据。
- UserResponse 也应该用它训练，因为它要学习 `state + action -> reward`。
- Replay buffer / offline RL / simulator rollout 都应基于 transition 数据。
- `04 HPN warmstart` 可以保留为 baseline 或可选初始化，但不应该再作为创新主线。

### 04 HPN warmstart 是否可以删

不建议从代码上真删，因为它是现有 HSRL baseline 的一部分，后续论文/汇报需要对比。

但在新文件夹里可以把它降级为：

```text
optional warmstart baseline
```

而不是核心创新。

需要注意的是：如果完全不做 warmstart，RL 从随机 actor 开始会很难。比较稳的实验设计是：

| 实验 | 作用 |
|---|---|
| HSRL + original warmstart | 复现 baseline |
| HSRL without warmstart | 看 warmstart 依赖 |
| 新 SID + no warmstart | 看新 SID 是否能支撑 RL |
| 新 SID + light warmstart | 折中方案 |
| 新 SID + regret pool | 我们的方法 |

### 当前推荐结论

数据切分要从“给 HPN 做监督 next item prediction”转成“给 RL 做 state-action-reward-next_state transition”。

这意味着：

- 不能只关心 target item 是否预测准确。
- 要关心 action 的 reward。
- 要保留 next_state。
- 要能支持 regret 的 delayed credit assignment。

### 需要和你确认

1. user-item episode 是按 `close_gap_seconds` 切，还是同一用户全局同 item 聚合？
   我建议不要全局聚合，建议保留时间窗口 episode。
2. unlike 是否一定视为 regret？如果用户 unlike 很久以前 like 的歌，是否仍回溯惩罚原推荐？
3. undislike 是正向 regret，还是只表示取消负反馈？
4. reward 是否允许负数？我建议允许 raw negative，但训练 reward 做 clip/scale。
5. 是否把 low play ratio 也当 implicit regret？docx 里提到了 engagement-based regret，我建议加入。

## 方向三：反悔池 / Regret-Aware Policy Intervention

### docx 2.4 的核心意思

我可以从 docx 读出 2.4 的文字，但公式在转换时丢失了，需要你后续确认原公式。

2.4 的核心是：

> 为每个用户维护一个滑动窗口 regret memory pool，里面存最近失败的 semantic paths 和对应 regret strength。HPN 在逐层生成 SID token 时，计算当前候选 token 的前缀路径和 regret pool 中失败路径的语义重叠，把这个重叠转成 penalty，作为 soft mask 加到原始 logits 上，降低模型再次走向失败语义路径的概率。

这可以理解为：

```text
用户曾经后悔/反悔过某类语义路径
=> 后续推荐时不要完全禁止这类路径
=> 但要降低它的生成概率
```

### 为什么这是创新点

原始 HSRL 只利用当前状态生成 SID，它没有显式记住“这个用户最近后悔过哪些语义路径”。反悔池的价值是：

- 把 unlike / undislike / low play ratio 等负面动态反馈变成可操作的 semantic memory。
- 不只是事后调整 reward，还能在下一次 action generation 时直接干预 policy。
- 因为 SID 是层级的，可以在粗粒度层面避免相似失败类别，而不是只屏蔽单个 item。

### 如何定义 regret memory pool

对每个用户维护：

```text
M_u = [(sid_path_j, phi_j, timestamp_j, regret_type_j, item_id_j), ...]
```

字段：

| 字段 | 说明 |
|---|---|
| `sid_path_j` | regret item 的 SID path |
| `phi_j` | regret strength |
| `timestamp_j` | regret 发生时间 |
| `regret_type_j` | unlike / undislike / low_play / dislike |
| `item_id_j` | 对应 item |

滑动窗口可以按两种方式：

- 最近 K 个 regret events。
- 最近 T 时间窗口。

建议先用：

```text
pool_size = 50
time_decay = exp(-delta_t / tau)
```

### regret strength 怎么算

初版可以先规则化：

```text
phi =
  a_unlike    * I(unlike)
  + a_dislike * I(dislike)
  + a_lowplay * max(0, play_threshold - play_ratio)
  - a_undislike * I(undislike)
```

但这里要小心：

- unlike 是从正反馈转负/撤销，通常是强 regret。
- dislike 是直接负反馈，不一定是 regret，但可作为 failed path。
- undislike 是撤销负反馈，可能代表“负面判断后悔”，它不应该和 unlike 同方向惩罚。
- low play ratio 是隐式 regret 或低满意度，但噪声大。

建议第一版只把 `unlike + dislike + low_play` 放进 negative regret pool；`undislike` 先作为单独正向修正，不直接加入同一个负池。

### policy intervention 怎么做

HPN 原本逐层输出：

```text
logits_l = head_l(context_l)
```

加入反悔池后：

```text
logits'_l = logits_l - eta * penalty_l
```

其中 `penalty_l` 来自当前 prefix 和 regret pool 里 failed SID paths 的重叠。

直觉：

- 如果当前前缀和某个 regret path 前几层高度重合，那么这一层继续走向对应 token 应被扣分。
- 层级越浅，影响越粗；层级越深，影响越具体。
- penalty 不应该是 hard mask，否则会过度屏蔽用户可能重新喜欢的内容。

### 和现有代码的关系

现有 `SIDPolicy_credit.py` 里已经有一个接口：

```python
get_sara_logits(feed_dict, pool_tokens=None, pool_phis=None)
```

它目前标注为 SARA/RAPI 扩展入口，baseline forward 不调用。里面已经有：

- `pool_tokens`
- `pool_phis`
- one-hot path matching
- level weights
- `logits_l = logits_l - sara_eta * penalty`

这说明反悔池创新不是从零开始，现有代码已经有一个雏形。真正要做的是：

1. 在数据层构造 per-user regret pool。
2. 在环境或 reader 里把当前用户的 pool 传给 actor。
3. 在 `apply_policy` 里调用 `get_sara_logits` 而不是普通 `forward`。
4. 训练/评估时更新 pool。
5. 做 ablation：无 pool / hard mask / soft mask / 不同 eta / 不同 pool size。

### 与 Retrospective Regret Credit Assignment 的关系

docx 里 2.3 是 retrospective credit assignment，2.4 是 policy intervention。两者可以分开做：

- 2.3 影响训练时 advantage / reward credit。
- 2.4 影响推理或 rollout 时 action generation。

建议实验拆成：

| 模型 | 说明 |
|---|---|
| HSRL baseline | 无 regret |

## 对当前想法的总体评价

### 我认为合理的部分

1. **从 SID 编码质量切入是对的。**
   HSRL 的核心依赖 semantic action space。如果 SID 质量差，后面 RL 很难补救。

2. **PCA + RQKMeans 是一个清晰、可解释、容易消融的改动。**
   它比直接说“RQVAE 更好”更适合做论文创新对照，因为它仍然保持 HSRL 的离散层级思想。

3. **数据集要从 next-item prediction 转向 transition/reward 是对的。**
   如果目标是 RL，最终不能只看推荐 item 是否等于用户下一次交互 item，而要看 reward 和长期收益。

4. **反悔池是最像论文创新的部分。**
   它能把 Yambda 独有的 unlike/undislike 信号和 HSRL 的 semantic path 结合起来。

### 我认为需要谨慎的部分

1. **不要过早否定 HPN warmstart。**
   它可能不是创新，但作为 baseline 或初始化仍有价值。完全删除会让 RL 训练难度变高。

2. **不要把 RQVAE 的 MSE 和 SID 局部性混成一个指标。**
   MSE 低不代表前缀共享好；二者都要报。

3. **不要全局合并同一用户同一 item 的所有事件。**
   这会损伤时间序列和 transition 语义。建议用时间窗口 episode。

4. **不要只用候选集排序证明 RL。**
   候选集排序是 next-item sanity check，RL 需要 total reward / depth / regret-rate 这类指标。

### 当前必须先解决的问题

1. `train rows > kept events` 的异常到底是不是脚本 bug 或旧文件累积。
2. RQVAE 全量 `dense_item2sid.npy` 是否能恢复并诊断。
3. reward 权重如何定，尤其 unlike/undislike 的方向和强度。
4. regret pool 中是否包含 dislike 和 low-play。
5. 是否保留 HPN warmstart 作为初始化，还是只保留为 baseline。
